Case Study · 10 Weeks · Lead Product Designer (UX/UI & Accessibility)

Blue Badge Digital Companion

Streamlining accessibility for 2.5 million users. Redesigned the Blue Badge renewal process to reduce application drop-off and achieve 100% WCAG 2.2 AA compliance — applying GDS patterns and the GOV.UK Design System throughout.

Timeline
10 Weeks
Role
Lead Product Designer (UX/UI & Accessibility)
Tools
Figma · GOV.UK Prototype Kit · Axe DevTools
Standards
GDS · GOV.UK Design System · WCAG 2.2 AA

The Problem

Applying for a Blue Badge is a high-stakes task for users with physical or cognitive impairments. The existing service was fragmented — requiring physical documents by post — leading to a 35% application rejection rate due to incorrect evidence submission. Users felt anxious and unsupported during the 8-week waiting period, and 15% relied on a family member to complete the form on their behalf.

01

Discovery & Insights

Audited the existing journey against WCAG 2.2 and conducted accessibility-focused user research. Key findings: users with motor impairments struggled with small hit zones on mobile; cognitive load was the biggest barrier with medical history and identity proof combined in a single exhausting flow; and the system had no accessible route for third-party (family member) submissions.

02

Ideation & Exploration

Applied the GDS Task List pattern to break the application into digestible, single-purpose steps — following the "one thing per page" principle throughout. Introduced a "Save and come back later" feature, recognising that users with chronic pain or fatigue cannot always complete a 20-minute form in a single sitting. Third-party submission flows were also scoped.

03

Design & Prototype

Built on the GOV.UK Design System for familiarity and trust. Implemented magic-link authentication to avoid cognitive function tests (WCAG 2.2 – 3.3.7). Ensured all interactive elements met the 24×24 CSS px minimum target size (WCAG 2.2 – 2.5.8). Customised CSS focus rings to high-contrast yellow/black — so keyboard-only users never lost their place in the flow.

04

Testing & Iteration

Conducted usability sessions with 5 participants using screen readers and switch access. Testing revealed the Upload button confused users with tremors — iterated to support both drag-and-drop and a traditional file picker (WCAG 2.2 Redundant Entry). Replaced a percentage-based progress bar with a GDS Task List after screen reader users found percentage progress too vague and non-actionable.

05

Implementation & Deliverables

Delivered a high-fidelity prototype ready for GDS Service Assessment. Key deliverables included: a service map linking digital touchpoints to physical badge delivery; an accessibility audit spreadsheet documenting passes across all 55 WCAG 2.1/2.2 AA criteria; and a documented GDS component library covering buttons, radios, error summaries, and task list patterns.

06

Accessibility Highlights

  • Magic-link login — no cognitive function tests (WCAG 2.2 3.3.7)
  • All interactive targets min. 24×24 CSS px (WCAG 2.2 2.5.8)
  • High-contrast yellow/black focus rings for keyboard navigation
  • Drag-and-drop + file picker redundancy (WCAG 2.2 Redundant Entry)
  • GDS Task List replacing vague percentage progress bars
  • Save & return flow for users with chronic fatigue or pain
  • 100% WCAG 2.2 AA validated via Axe DevTools + manual testing
