Overview

The Blue Badge scheme enables people with severe mobility issues or hidden disabilities to park closer to their destinations. Despite serving over 2.5 million badge holders in the UK, the renewal journey was fragmented, stressful, and inaccessible — the very population it aimed to support was most likely to struggle with it.

I was brought in as Lead Product Designer to redesign the digital application journey, with hard constraints: full conformance with the GOV.UK Service Standard, WCAG 2.2 AA compliance, and a service that genuinely worked for users with physical, cognitive, and sensory impairments.

"I gave up halfway through. There were too many pages asking for too much at once. My hands were shaking by the end and I'd lost everything I'd typed."
— Research participant, applicant with rheumatoid arthritis

Discovery: The Problem

Applying for a Blue Badge is a high-stakes task for users with physical or cognitive impairments. The existing service was fragmented, requiring users to mail physical documents — leading to a 35% application rejection rate due to "incorrect evidence." Users felt anxious and unsupported during the 8-week waiting period with no visibility of their application status.

Core Problems Identified

  • Physical document submission created barriers for users with limited mobility or no printer access
  • Single long form caused fatigue — no way to save and return, abandonment spiked after 8 minutes
  • Medical history and identity proof were requested in the same exhausting flow
  • No application status tracking during the 8-week processing window
  • 15% of users relied on family members to complete the form but the system had no third-party permission flow

WCAG 2.2 Audit Findings

Hit Zone Failures
Interactive elements fell below the 24×24 CSS pixel minimum (WCAG 2.5.8). Users with motor impairments consistently triggered wrong buttons on mobile.
Cognitive Overload
The existing flow asked for medical history, identity evidence, and personal details simultaneously — a direct conflict with the "one thing per page" GDS principle.
Assisted Digital Gap
No mechanism for authorised third-party submission. 15% of users were forced to share personal login credentials with family members — a safeguarding risk.
02

Insights: Defining the Accessibility Challenge

I conducted qualitative research sessions with 8 participants across three disability categories — motor impairment, cognitive impairment, and low vision — alongside interviews with Local Authority caseworkers who process applications daily.

"I need to stop halfway through. My condition means I can't concentrate for more than ten minutes. When the session timed out, I lost everything. I just cried."
— Research participant, applicant with MS

Design Questions

  • How might we break a complex application into manageable chunks without losing user context?
  • How might we authenticate users without relying on cognitive function tests that disadvantage our core audience?
  • How might we support third-party assisted digital journeys without creating safeguarding risks?
  • How might we give applicants confidence and status visibility during an 8-week wait?

Problem Statement

Blue Badge applicants — often living with the very conditions that qualify them for the scheme — need a digital service that adapts to their pace, cognitive capacity, and assistive technology. The current service fails at the first interaction and abandons users mid-journey with no way back.

03

Ideation: GDS Patterns & Inclusive Flows

I anchored the redesign in two GDS principles: "One thing per page" and the Task List pattern. These aren't just best practice — for users with cognitive impairments, they are a functional accessibility requirement.

The Strategy

Task List Pattern
Using the GDS Task List component, the application was broken into 5 named sections. Users could see exactly where they were, what was complete, and what remained — reducing anxiety significantly.
Save and Return
Recognising that users with chronic pain or fatigue may not finish a 20-minute form in one sitting, I designed a persistent save-and-return flow accessible with just an email link — no password required.
Third-Party Permissions
A new delegated access flow allowed a named family member or carer to complete and submit on behalf of the applicant, with full audit trail — removing the credential-sharing safeguarding risk.

Design: WCAG 2.2 AA Implementation

I built the prototype in the GOV.UK Prototype Kit using the GOV.UK Design System component library. Every design decision was mapped directly to a WCAG 2.2 criterion, ensuring evidence of compliance was built into the design process — not retrofitted.

WCAG 3.3.7

Accessible Authentication (Minimum): I designed a magic link login flow that avoids "cognitive function tests" — no remembered passwords, no CAPTCHA. The user enters their email address once; a time-limited sign-in link is sent to their inbox. This directly addresses the cognitive barriers faced by users with dementia or memory impairment.

WCAG 2.5.8

Target Size (Minimum): All interactive elements — including the "Upload Evidence" button, radio buttons, and navigation links — were sized to a minimum of 24×24 CSS pixels with sufficient surrounding space to prevent accidental triggers. Motor-impaired users with tremors were the primary beneficiary.

WCAG 2.4.11

Focus Appearance: I customised CSS focus rings to use a high-contrast yellow (#FFDD00) on black outline — the GOV.UK standard — ensuring keyboard-only users and switch access users always had a clear, unmissable focus indicator on every interactive element.

WCAG 3.3.7

Redundant Entry: The "Upload Evidence" flow was designed to offer both drag-and-drop and a traditional file picker. Users with tremors who struggle with precise drag interactions could use the file picker instead — providing an equivalent, accessible alternative without degrading the experience.

Focus State Design — Before & After

Before — Fails WCAG 2.4.11

Browser default focus ring

The default browser focus outline was overridden with outline: none in the original CSS — completely invisible to keyboard users and failing WCAG 2.4.11 at any level.

After — Passes WCAG 2.4.11

High-contrast custom focus ring

3px solid yellow (#FFDD00) outline with 2px offset on a dark background — meeting the minimum 3:1 contrast ratio change requirement and matching GOV.UK Design System standards.

Component Library Delivered

All components were documented in Figma with accessibility annotations covering ARIA roles, focus order, error states, and keyboard interaction patterns — ready for developer handover.

Buttons
Primary, secondary, warning variants. Min 24px target. High-contrast focus states documented.
Form Inputs
Text inputs, radios, checkboxes, date inputs. Error states with inline messages and summary links.
Error Summary
GDS Error Summary pattern — focus moves to summary on submit. Each error links to the relevant field.
File Upload
Drag-and-drop with accessible fallback file picker. Upload progress and success confirmation states.
05

Testing: Assistive Technology Sessions

I conducted moderated usability sessions with 5 participants using assistive technologies across two rounds. Participants used NVDA (screen reader), VoiceOver (iOS), and Switch Access (Android) — covering the primary AT used by the Blue Badge population.

Testing Scenarios

  • Complete sections 1 and 2 of the application using keyboard-only navigation
  • Upload a PDF document using the evidence upload component
  • Save the application and return via magic link on a different device
  • Recover from a form validation error using only a screen reader

Key Pivots from Testing

Original Design

Progress Bar

A percentage-based progress bar (e.g. "60% complete") was used to show application progress. Screen reader users found the percentage meaningless without section context — "60% of what?"

Iteration

Task List Component

Replaced with the GDS Task List pattern. Each section is named, has a clear status (Not started / In progress / Completed), and screen readers announce the full context on focus. Comprehension improved from 40% → 95%.

Original Design

Upload Button Only

A single "Upload file" button. Users with tremors consistently triggered adjacent elements accidentally, and those using Switch Access found the small target area with single button caused repeated errors.

Iteration

Drag & Drop + File Picker

Added both a drag-and-drop zone and a traditional file picker button. Switch Access users adopted the picker; pointer users preferred drag-and-drop. Both paths lead to the same outcome — meeting WCAG 3.3.7 Redundant Entry.

Delivery: GDS Assessment Readiness

The final output was a high-fidelity prototype built in the GOV.UK Prototype Kit, validated by automated Axe DevTools scans and manual assistive technology testing — ready for GDS Service Assessment at Beta stage.

Key Deliverables

Service Map
End-to-end map of digital touchpoints, physical processes, and Local Authority back-office integrations across the full badge lifecycle.
Accessibility Audit
Spreadsheet documenting pass/fail/note status for all 55 WCAG 2.1 and 2.2 AA criteria, with evidence screenshots and remediation notes for each.
Component Library
Figma library of all GDS patterns used, with accessibility annotations covering ARIA roles, focus order, keyboard interactions, and error states.

Accessibility Audit — Selected WCAG 2.2 Criteria

Criterion Level Description Status Evidence / Notes
1.4.3 AA Contrast (Minimum) Pass All body text ≥ 4.5:1. CTA buttons ≥ 7:1. Verified with Axe + manual checks.
1.4.11 AA Non-text Contrast Pass Form input borders, focus indicators, and UI components all meet 3:1 minimum.
2.4.11 AA Focus Appearance Pass Custom yellow (#FFDD00) 3px focus ring with 2px offset. Meets 3:1 contrast change requirement.
2.5.8 AA Target Size (Minimum) Pass All interactive targets ≥ 24×24 CSS px with exception-free spacing. Verified on mobile viewport.
3.3.7 AA Redundant Entry Pass File upload provides both drag-and-drop and file picker paths. "Check your answers" pre-populates all fields.
3.3.8 AA Accessible Authentication (Minimum) Pass Magic link login avoids all cognitive function tests. No CAPTCHA or password memorisation required.
1.3.1 A Info and Relationships Pass All form labels programmatically associated. Error messages linked via aria-describedby. Headings in correct order.
2.1.1 A Keyboard Pass All functionality operable via keyboard. Drag-and-drop has keyboard-accessible alternative. No keyboard traps.
2.4.3 A Focus Order Pass Focus follows DOM order throughout. Modal dialogs trap focus correctly and return focus on close.
3.3.1 A Error Identification Pass GDS Error Summary component used. Focus moves to summary on submit. Each error links to the field.
2.5.3 A Label in Name Note Icon-only buttons in the file list use aria-label matching visible text. Pending dev implementation review.

Key Learnings

Accessibility is design, not QA
WCAG criteria mapped to design decisions from day one — not audited afterwards. This is the only way to build it in rather than bolt it on.
One thing per page works
The GDS "one thing per page" principle isn't just usability guidance — for users with cognitive impairments, it's a functional accessibility requirement.
Save & return is essential
Designing for chronic fatigue and pain means designing for users who may not complete a task in one sitting. Persistence and session recovery are core, not nice-to-have.
Test with real AT users
Automated tools caught ~30% of issues. The task list vs progress bar pivot — the most impactful change — only emerged from manual testing with screen reader users.
